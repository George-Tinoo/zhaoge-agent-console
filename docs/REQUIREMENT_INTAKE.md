## 0. 基础场景与设备形态

### 0.5 设备连接与发现（已补充）

**连接方式**（优先级从高到低）：
1. **扫码绑定** — 扫描设备二维码完成绑定
2. **蓝牙发现** — App 自动发现附近可配对的设备
3. **局域网查找** — 同一 WiFi 下自动发现设备

### 0.6 模型选择与参数配置（已补充）

**模型选择入口**：
- 聊天界面随时切换模型
- 用户设置中统一配置默认模型
- **AI 原生方式**：直接在聊天中通过自然语言更改（如"换 GPT-5.5"、"用轻量级模型"）

**参数配置原则**：
- **全部参数对用户暴露**，用户有权了解全部内容
- temperature、max_tokens、top_p 等高级参数可配置
- 提供默认配置，但用户可完全自定义

### 0.7 设备状态展示（已补充）

需实时显示：
- 在线/离线状态
- 存储空间剩余容量（**快满时红色百分比警告**）
- 实时负载情况（CPU/GPU/内存）
- 设备运算状态 — 是否正在忙碌、能否响应新请求

### 0.8 多设备切换与离线处理（已补充）

**多设备热切换**：
- 保持多个绑定的设备同时在线
- 用户可随时选择使用哪个设备的算力源
- 无缝切换，不中断当前会话

**设备离线处理**：
- 当前连接设备离线时，**自动切换到云端算力兜底**
- 同时提示用户："之前连接的设备已离线，功能受限"
- 明确告知用户当前使用的是云端算力，部分本地功能不可用

**存储空间预警**：
- 设备存储快满时，**通过对话框内容交流提醒用户**（如 AI 主动说："大王，我的存储快满了，需要清理一下"）
- 设备状态栏显示存储百分比，**快满时变红色警告**
- 提供一键清理或手动管理选项

### 0.1 三类连接对象

本 App 可能连接三类对象：

1. **云端智能体服务 / 端脑云**
   - 提供云端智能体服务。
   - 提供云端大语言模型算力。
   - 提供注册、登录、账号验证等账户系统能力。

2. **龙虾派**
   - 本地硬件。
   - 可安装类似 OpenClaw、Hermes Agent、VCP 等智能体。
   - 具备约 32G 到 512G 本地存储。
   - 连接方式：蓝牙、同一 WiFi 网络、扫码配对。
   - 本地智能体也可调度端脑云的云端算力。
   - App 内需要能配置云端大语言模型服务，供龙虾派侧智能体调用。

3. **AI NPC**
   - 连接方式与龙虾派类似：蓝牙、同一 WiFi 网络、扫码配对。
   - 与龙虾派区别：AI NPC 可安装本地 120B 大语言模型，具备本地大模型算力。
   - 具备约 100T 存储。
   - 也可调度端脑云的云端算力。
   - App 内同样需要配置云端大语言模型服务。

### 0.2 用户注册与登录

- 用户使用 App 前需要注册/登录。
- 登录/注册依赖端脑云账户系统。
- 登录：App 将用户登录请求发送给端脑云，验证通过后允许登录。
- 注册：若用户没有账户，App 调用端脑云注册能力，帮助用户通过手机号完成注册。

### 0.3 端脑云账号与设备关系（已澄清）

**账号规则**：
- 端脑云是唯一主账号，允许设备离线使用
- 一个账号可绑定多台设备，但**同时只能使用一台**设备
- 可以在多个绑定的算力源（设备）之间进行切换
- 一个设备只能绑定一个账号，**不能绑定多个账号**

**权限管理**：
- 目前只有管理员最高权限
- 暂不开放其他级别的权限

**产品定位**：
- 每个人需要有自己的 AI 设备
- 每个人需要有自己的 AI 助手交互 App

### 0.4 仍待补充问题

- 端脑云是否不仅提供账号系统，还提供设备绑定、云端智能体列表、模型配置、订阅/计费、推送等能力？
- 龙虾派和 AI NPC 是否都必须绑定到端脑云账号名下？是否允许离线/局域网游客模式？
- 蓝牙/WiFi/扫码配对三种方式的优先级、失败兜底、配对安全机制待定义。
- 云端算力配置是用户级、设备级、智能体级，还是会话级？
- 本地算力与云端算力的调度策略待定义：手动选择、自动路由、成本/速度/隐私优先。

## 3. 首页设计（已补充）

### 3.1 首次使用引导流程（新手引导）

**登录注册阶段**：
- 打开 App 先完成注册/登录
- 登录方式可传统（调端脑云账户），但界面要**AI 风格**，不照搬 Web2

**首次引导对话**（相互了解）：
1. **询问用户称呼** — "您希望我如何称呼您？"
2. **询问 Agent 称呼** — "您想怎么称呼您的智能体？"
3. **告知可调整** — 说明这些设定以后都可以在聊天中随时更改
4. **介绍功能范围** — 告诉用户可以通过对话框实现：
   - 创造、写作、绘画
   - 代码、设计、财务、法律
   - 健康医疗、学习、生活助手等
   - 所有需求只需通过对话交流即可实现
5. **收集个人信息**（可选跳过）：
   - 出生年月日、民族、性别
   - 住宅、饮食等生活信息
   - 聊得越细越好，帮助 AI 更懂用户

### 3.2 正式首页（每日首页）

**开启条件**：完成首次引导后，每次登录进入

**首页内容组成**：

#### ① 个人运势信息区（传统文化 + AI 结合）
结合以下信息生成专属个人引导：
- 用户已提供的个人信息
- 当天地理位置
- 当天黄历信息
- 当天年月日时的八字
- 八字匹配测算

**展示内容**：
- 吉凶方位
- 形象建议
- 情感交流建议
- 事业顺利度
- 幸运颜色
- 幸运数字
- 幸运方位
- 等一系列专属个人引导

#### ② "开启新的一天" 按钮
- 点击后进入当天的主要功能界面
- **每日重置**：每天凌晨 6:00 后重置当天信息

#### ③ 形象设计区
- **用户形象**：可上传照片为自己设计 AI 形象
- **Agent 形象**：可上传照片为 App/Agent 设计形象
- **多 Agent 头像**：不同 Agent 使用不同头像聊天

#### ④ Agent 角色系统
**角色来源**：
- **用户自定义**：用户给 Agent 设定身份性格
- **随机获取**：从预设 100 个角色中随机获得
  - 角色库覆盖：中国、美国、欧洲、日本等不同文化地区
  - 角色来源：各种火爆游戏中的经典角色
  - 继承真实身份性格特点与用户交流

**角色调整**：
- 后续聊天中可随时调整 Agent 角色设定
- 形象 + 性格 + 声音 都可自定义

#### ⑤ 新功能推送区
**功能**：
- 版本更新、新功能上线时首页展示
- 以对话窗口形式介绍新功能
- 包含：新功能是什么、怎么用、使用教学
- 让用户与最新功能保持同步，快速喜欢上新功能

### 3.3 首页设计理念

- **AI 原生**：不是 Web2 的传统列表式首页
- **交互式引导**：通过对话完成相互认知
- **个性化运势**：传统文化（黄历八字）+ AI 个性化结合
- **角色趣味性**：形象 + 角色扮演增加情感连接
- **功能发现**：新功能通过对话自然推送，而非弹窗广告 

## 4. 聊天体验设计（已补充）

### 4.1 聊天框布局（消息展示区域）

**基础布局**：
- 智能体回复在**左侧**，用户消息在**右侧**
- 所有文字**可被选取和复制**

**代码渲染**：
- 智能体回复中包含可执行代码时，在消息下方**直接渲染 HTML5 效果**
- 聊天框支持 HTML5 内容展示和交互

**语音播报**：
- AI 可设定对我说文字或发语音
- **默认发送文字**，用户可要求 AI 改成发语音
- 用户可随时和 AI 交流，让它切换回复方式

**时间记录**：
- 聊天框记录每次聊天的时间戳

**消息操作**：
- **长按对方消息**可进行定向回复（引用回复）

**多媒体处理**：
- **图片**：可点击放大浏览、可存储保存
- **音频**：可播放收听、可转文字查看
- **视频**：可播放、可存储，播放时**全屏放大播放**

**导航功能**：
- **一键到底**：快速回到底部最新消息
- **悬浮导航按钮**：聊天内容过长时，右侧显示悬浮按钮
  - 通过上下滑动快速定位到聊天任意位置（头部、中间、底部）
  - 按钮可隐藏/呼出

**聊天记录管理**：
- 每个聊天**独立记录**
- **界面左上角可展开聊天标题列表**
- 切换标题自动切换到对应聊天记录
- 根据已记录的**上下文继续后续沟通**

**未来扩展（人与人/Agent 间聊天）**：
- 当前：用户 ↔ 自己的 Agent
- 未来支持四种模式：
  1. 自己 ↔ 自己的 Agent
  2. 自己 ↔ 人类好友
  3. 自己 ↔ 别人的 Agent
  4. 自己的 Agent ↔ 别人的 Agent

### 4.2 输入框设计

**附件添加**：
- **手机本地**：照片库选取、相机拍照
- **连接设备**：可添加设备上的文件（照片、视频、PDF、压缩包等**所有文件格式**）

**输入方式**：
- **打字输入**：常规键盘输入
- **语音输入**：**长按输入框**切换为语音输入状态
  - 长按状态下说话自动识别
  - 停止说话后自动转为文字填入输入框
  - 用户点击发送后才发送

**快捷功能按钮（输入框右侧）**：
- **相机按钮**：唤起手机相机
  - 支持实时拍照、录像
  - 完成后可直接发送到聊天框
- **录音按钮**：唤起录音功能
  - 实时录音
  - 完成后可将录音文件发送至聊天框 

## 5. 多 Agent 协作设计（已补充）

### 5.1 Agent 基础配置

每个 Agent 需要配置三项基础设定：
1. **角色定义** — Agent 的身份、性格、形象
2. **大模型服务** — 支持该 Agent 的后端 LLM（可不同 Agent 用不同模型）
3. **工作类型** — 该 Agent 适合完成的任务类型：
   - 记录/记忆类工作
   - 数据分析
   - 代码开发
   - 绘画创作
   - 文案写作
   - 更多可扩展类型...

**Agent 来源**：
- 云端镜像（OpenClaw、Hermes Agent 等）
- 本地设备（龙虾派、AI NPC 上安装的 Agent）
- 不同 Agent 可用相同或不同的 LLM 后端

### 5.2 主管 Agent 机制（大内总管）

**角色定位**：
- 用户的**主要交流对象**
- 负责将用户需求拆解为不同类型的工作任务
- 组织协调不同 Agent 分工协作
- 汇总各 Agent 工作结果，形成完整结论给用户

**工作流程**：
1. 用户向主管 Agent 提出需求
2. 主管分析需求，拆解为子任务
3. **先向用户汇报组织方案和计划**，等待用户确认
4. 用户确认后，分配给各 specialist Agent 执行
5. 收集结果，汇总整理，反馈给用户

**特殊情况**：
- 若只有一个 Agent，该 Agent 既是主管也是执行者
- 主管 Agent 也可直接处理具体事务，不限于管理工作

### 5.3 直接唤起与独立对话

**唤起方式**：
- 用户直接叫 Agent 的名字（如"子牙，帮我看看这个代码"）
- 被叫到的 Agent 立即唤起，通过 App 独立聊天窗口与用户交流
- 每个 Agent 拥有**独立的聊天频道**

**使用场景**：
- 用户可直接与任一 Agent 单独交流
- 也可只和主管 Agent 交流，由主管协调其他人

### 5.4 动态任务分配机制

**新任务类型处理流程**：

1. **发现问题**：
   - 用户提出的需求涉及未指定负责 Agent 的功能（如需要绘画 Agent，但目前只有文案和代码 Agent）
   - 主管 Agent 报告用户："目前没有负责此功能的 Agent"

2. **内部讨论**：
   - 主管组织**所有现有 Agent 内部讨论**
   - 讨论内容：
     - 现有 Agent 谁更适合承接这个新任务？
     - 如果都不适合，需要创建什么样的新 Agent？
     - 新 Agent 应该用什么大模型/后端服务支持？

3. **用户决策**：
   - 讨论结束后，用户可要求主管：
     - 创建一个新的 Agent 专门负责此类任务
     - 或分配给某个现有 Agent 兼管

4. **后续执行**：
   - 新 Agent 创建/分配后，自动纳入主管的调度体系
   - 未来同类任务直接分配给该 Agent

### 5.5 协作示例

**示例一：单 Agent 模式**
- 只有 1 个 Agent
- 该 Agent 既是主管，也负责所有具体工作
- 用户只与这一个 Agent 交流

**示例二：妲己 + 子牙 双 Agent 模式**
- **妲己**担任主管 Agent
- 负责：与用户交流、任务拆解、协调分工、结果汇总
- **子牙**担任 specialist Agent
- 负责：复杂架构、代码开发、技术分析
- 妲己将子牙的工作结果汇总，给大王最终结论

### 5.6 权限与确认机制

- 主管 Agent **必须先汇报计划，经用户确认后才能执行**
- 用户始终掌握最终决策权
- 可随时打断、修改方案、或直接指挥某个 Agent

## 7. 文件与知识库设计（已补充）

### 7.1 双维度沉淀体系

内容以**两个维度**呈现在 App 中，让用户有清晰感知：

| 维度 | 项目（Project） | 知识（Knowledge） |
|------|----------------|------------------|
| **分类方式** | 按任务/目标/事项分类 | 按主题/领域分类 |
| **内容来源** | 特定项目的沟通内容 | 跨项目的同类知识 |
| **结构** | 动态列表，Agent 自动归类 | 主题目录，动态增加 |
| **存储内容** | 聊天记录、文件索引、图片、视频、媒体 | 领域知识、经验沉淀、参考资料 |

### 7.2 项目库（Project Library）

**入口位置**：聊天界面**右上角项目按钮**

**功能特性**：
- 所有沟通内容**自动分类规整**为项目列表
- 每个项目包含：
  - 项目相关所有聊天记录
  - 自动分类整理的内容
  - 文件索引、图片、视频、多媒体文件位置
- **动态增长**：Agent 自动识别新事项并创建/归类项目
- **无法归类的内容**：统一放入"杂项"或等待 Agent 后续整理

**使用场景**：
- "装修项目" — 所有装修沟通、设计图、预算表
- "小说创作" — 章节讨论、角色设定、剧情大纲
- "代码开发" — 需求讨论、代码文件、测试记录

### 7.3 知识库（Knowledge Base）

**定义**：按**主题/领域**组织的内容，可能跨多个项目

**功能特性**：
- 同方向、同领域的知识归入同一目录
- **动态增加**：Agent 识别新知识点自动扩展目录
- 支持多层级目录结构
- 存储文件索引和多媒体资源位置

**使用场景**：
- "AI 技术知识" — 跨项目的代码经验、架构思路
- "育儿知识" — 宝宝成长相关信息（跨聊天记录）
- "设计规范" — 装修、UI 等设计原则沉淀

### 7.4 内容沉淀机制

**来源覆盖**：
- ✅ 用户与 Agent 的沟通
- ✅ 用户与其他用户的沟通
- ✅ Agent 与 Agent 之间的沟通
- ✅ 所有沟通中的文件、图片、视频、媒体

**存储方式**：
- 实际文件存储在连接的设备/云端（龙虾派、AI NPC、端脑云）
- App 中存储**文件索引**和**元数据**
- 多媒体文件位置统一记录，方便调用

### 7.5 记忆共享机制

**核心能力**：
- 即使**换不同的 Agent 沟通**，也能访问之前沉淀的全部记忆
- 所有 Agent 共享项目和知识库的内容
- 有结构的内容沉淀支持长期工作和学习
- 新 Agent 加入时可快速了解历史背景

### 7.6 与传统文件管理的区别

| 传统方式 | 朝歌方式 |
|---------|---------|
| 用户手动创建文件夹 | Agent 自动识别、归类、创建 |
| 文件存在某个固定路径 | 文件存在连接的设备，App 存索引 |
| 跨 App/设备无法共享 | 所有 Agent 和用户共享同一知识库 |
| 搜索靠文件名 | 语义检索，理解内容主题 | 

## 8. 主动提醒与任务系统（已补充）

### 8.1 推送机制（Agent 主动触达）

**核心能力**：
- 突破当前 App **无推送**的限制
- **Agent 可主动给用户发消息**，而非被动等待用户提问
- 用户规划的任何事项，Agent 可在设定时间**主动提醒和反馈汇报**

**价值**：

**核心定位**：

- **所有用户必须拥有移动端 App**

**已覆盖功能**（前文已述）：
- 选中复制粘贴

**文件存储**：

### 9.2 桌面端（生产力扩展）

**核心定位**：

**登录方式**：
1. **首选**：移动端 App 扫码登录（保证一致性）
2. **备用**：手机号 + 验证码/密码登录

**桌面端专属增强**：

**功能一致性**：

**核心输入方式**：

**语音输出选项**：
- 用户可要求 Agent "用语音回复"

## 11. 权限与安全设计（已补充）

### 11.1 当前权限模型（单用户模式）

**现状**：
- 当前设计为**单一主人**使用自己的 App
- 用户通过账号登录后，**可执行所有操作**
- 对自己的设备、文件、Agent 拥有完全控制权

### 11.2 未来扩展：跨实体通信权限（严格管控）

**适用场景**：
- 我的 Agent ↔ 其他人类用户
- 我的 Agent ↔ 其他用户的 Agent
- 涉及**对外信息传递**和**跨域动作执行**

#### 信息传递权限规则

**基本原则**：
- **任何信息传递给外部实体，必须得到用户审批同意**
- 无论是传给其他人类还是其他 Agent

**审批模式**：
- **每次审批**：每次传递都需用户确认（最高安全）
- **批量审批**：用户可确认"同类内容后续是否仍需审批"
  - 例："是否允许向该用户持续分享项目进度，无需每次确认？"
  - 用户可随时撤销批量授权

#### 动作执行权限规则

**基本原则**：
- **每一次动作执行都需要得到用户认可**
- **无例外**（即使是看似无害的操作）

**执行前确认**：
- 动作内容预览
- 影响范围说明
- 用户明确点击"同意执行"后方可继续

#### 高危行为标记与提醒

**自动标记为高危的场景**：
1. **涉及个人隐私信息**
   - 身份证号、住址、联系方式
   - 家庭成员信息
   - 财务账户信息
   - 健康医疗记录
   - 其他敏感个人数据

2. **涉及资金安全**
   - 任何支付、转账、消费行为
   - 账户密码、支付密码相关
   - 投资、理财操作
   - 合同签署、资金授权

**高危行为处理**：
- **醒目标识**：红色警告、特殊图标、明显提示
- **重点提醒**：强制用户审查内容细节
- **二次确认**：必须明确输入"确认"或生物识别验证
- **记录留痕**：所有高危操作记录日志，便于追溯

### 11.3 权限审批界面设计

**审批弹窗要素**：
- 谁在请求？（Agent 名称 + 头像）
- 请求类型？（信息传递 / 动作执行）
- 具体内容？（展示完整内容预览）
- 传递给谁？（目标人类/Agent 身份）
- 风险等级？（普通 / 高风险）
- 操作按钮？（同意 / 拒绝 / 修改后同意）

**批量授权管理**：
- 设置中可查看所有已授权的批量权限
- 可随时撤销特定授权
- 定期提醒用户审查现有权限

### 11.4 安全原则总结

| 场景 | 权限要求 |
|------|---------|
| 用户自己的设备/文件 | 完全控制，无需额外确认 |
| 信息传递给外部 | **必须审批**，可选批量授权 |
| 动作执行（任何） | **必须审批**，无例外 |
| 涉及隐私/资金 | **高危标记**，重点提醒，二次确认 |

### 12.5 长远拓展一：向内 - 深度用户理解（核心技能）

**目标**：
App 需要越来越了解用户，建立更深化的交互和更长久的关系。

**已覆盖维度**（玄学命理）：
- 八字
- 星座
- 紫微斗数
- 塔罗牌运势

**待扩展维度**（科学生命科学）：
- **基础生理**：身高、体重、年龄
- **情绪心理**：通过日常沟通判断情绪状态、压力水平
- **医疗健康**：
  - 身体健康信息
  - 病史、治愈情况
  - 家族病史
  - 体检报告（支持导入）
  - 基因报告（支持导入）
- **运动健康**：运动习惯、体能数据

**信息收集方式**：
- 用户主动导入（医疗报告、体检报告、基因报告）
- Agent 每天主动询问健康状况
- 通过日常对话自然收集

**价值呈现**：
- 在每日首页、信息沟通、汇报、提醒中逐渐应用这些数据
- **不需要用户自己构建**，这是 App 长期成长的**固定技能**

### 12.6 长远拓展二：向外 - 去中间商（重构商业）

**核心理念**：
去掉所有中间商，让信息交流价值**最直接、最快速、最优化**。

**实现方式**：
1. **用户间沟通前置过滤**：
   - 用户间沟通先通过各自的 Agent 进行
   - Agent 之间进行快速信息沟通、信息过滤
   - 提高人类交流的信息准确度和可信度

2. **去掉中间商平台**：
   - **招聘网站** → Agent 直接展示主人能力、获取需求、接单
   - **电商网站** → Agent 直接向其他 Agent 提出需求、提出 Offer
   - **服务网站** → Agent 直接匹配服务能力与需求

3. **Agent 商业行为**：
   - 向其他用户的 Agent 直接提出需求
   - 主动展示自己主人的能力
   - 获取需求、获取 Offer、接单
   - 完成商业闭环

**App 集成**：
- 快速操作其他辅助 App（电商、外卖、导航等）
- 全面拓展生活和工作场景
